## 들어가며

DB를 배울 때는 항상 외래키(FK)를 걸어서 참조 무결성을 보장하라고 배웁니다.  
그런데 실무 코드를 보다 보면 FK가 없는 테이블이 생각보다 많습니다.

> "이거 FK 왜 없어요?"  
> "응, 우리는 안 써."

이 대화, 신입 때 한 번쯤 경험해보셨을 겁니다.  
왜 실무에서는 FK를 안 걸까요? 맹목적으로 따라가도 되는 걸까요?  
이번 글에서 그 이유와 판단 기준을 정리해보겠습니다.

---

## DB 연관관계란?

DB에서 연관관계는 크게 두 가지 레벨로 나눌 수 있습니다.

**물리적 연관관계 (FK 제약)**  
DDL 수준에서 `FOREIGN KEY` 제약을 걸어 DB가 직접 참조 무결성을 강제합니다.

```sql
ALTER TABLE orders
ADD CONSTRAINT fk_orders_user
FOREIGN KEY (user_id) REFERENCES users(id);
```

**논리적 연관관계**  
FK 제약 없이, 애플리케이션 코드(ORM, 쿼리)에서만 관계를 표현합니다.  
DB 입장에선 그냥 일반 컬럼입니다.

---

## FK를 걸었을 때의 장단점

### 장점

- **데이터 무결성 DB 레벨 보장**: 잘못된 참조 데이터를 원천 차단합니다.
- **명시적인 관계**: 스키마만 봐도 테이블 간 관계를 즉시 파악할 수 있습니다.
- **쿼리 최적화 힌트**: 일부 DB는 FK 정보를 쿼리 플래너에 활용합니다.

### 단점

- **성능 오버헤드**: INSERT/UPDATE/DELETE마다 FK 유효성 체크가 발생합니다. 트래픽이 많을수록 부담이 커집니다.
- **삽입/삭제 순서 제약**: 부모 레코드가 먼저 있어야 자식 레코드를 삽입할 수 있습니다. 배치 처리나 마이그레이션이 복잡해집니다.
- **샤딩/MSA 환경에서 불가**: DB가 분리되면 cross-DB FK는 지원되지 않습니다.
- **테스트 데이터 설정 불편**: 테스트마다 부모 데이터를 먼저 세팅해야 하는 번거로움이 있습니다.

---

## FK를 안 걸었을 때의 장단점

### 장점

- **스케일 아웃/샤딩 유연**: DB를 자유롭게 분리하고 확장할 수 있습니다.
- **MSA 환경 적합**: 서비스 간 DB 독립성을 유지할 수 있습니다.
- **배포/마이그레이션 자유**: 테이블 순서나 의존관계를 신경 쓸 필요가 없습니다.
- **성능 이점**: FK 체크 비용이 없습니다.

### 단점

- **무결성 책임이 앱으로**: 개발자가 코드에서 직접 검증해야 합니다.
- **고아 데이터 위험**: 참조하는 부모가 삭제됐는데 자식 데이터가 남을 수 있습니다.
- **스키마 가독성 저하**: 관계를 코드나 문서를 봐야 파악할 수 있습니다.

---

## 실무 판단 기준

상황에 따라 다릅니다. 아래 기준을 참고해서 판단해 보세요.

| 상황 | 권장 방향 |
|------|-----------|
| 소규모 서비스, 단일 DB | FK 사용 가능 |
| 대용량 트래픽, 성능 민감 | FK 제거 고려 |
| 마이크로서비스 아키텍처 | FK 제거 (사실상 필수) |
| 로그성/이력성 테이블 | FK 불필요 |
| 금융/결제 도메인 | FK 또는 보상 트랜잭션으로 무결성 보완 |

## 애플리케이션 레벨에서의 연관관계

FK를 DB에 안 걸더라도, 애플리케이션에서는 연관관계를 표현할 수 있습니다.

### JPA 사용 시

JPA에서는 `@ForeignKey(ConstraintMode.NO_CONSTRAINT)`를 사용하면 FK 제약 없이 연관관계 매핑만 유지할 수 있습니다.

```kotlin
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(
    name = "user_id",
    foreignKey = ForeignKey(ConstraintMode.NO_CONSTRAINT) // DDL에 FK 생성 안 함
)
val user: User? = null
```

이렇게 하면 JPA의 연관관계 탐색, 지연 로딩 등의 기능은 그대로 쓰면서 DB에는 FK 제약이 없는 상태가 됩니다.

### jOOQ 사용 시

jOOQ는 연관관계를 ORM처럼 표현하지 않습니다. 필요한 관계는 JOIN 쿼리로 직접 명시합니다.

```kotlin
dslContext
    .select(ORDERS.ID, USERS.NAME)
    .from(ORDERS)
    .join(USERS).on(ORDERS.USER_ID.eq(USERS.ID))
    .fetch()
```

이 방식은 관계가 쿼리에 명시적으로 드러나서, FK 없이도 의도가 명확합니다.

## 현실적인 절충안

요즘 많은 팀이 채택하는 현실적인 방식은 이렇습니다.

> **DB에는 FK를 걸지 않고, 애플리케이션에서 검증 + 주기적인 무결성 검사 배치를 운영한다.**

1. **앱 레벨 검증**: 서비스 로직에서 참조 대상 존재 여부를 확인하고 처리합니다.
2. **무결성 검사 배치**: 주기적으로 고아 데이터나 불일치 데이터를 감지하는 배치를 돌립니다.
3. **모니터링/알림**: 이상 감지 시 알림을 보내 빠르게 대응합니다.

Spring Batch를 쓴다면 이런 무결성 검사 배치를 구성하기에 딱 좋은 환경입니다.

---

## 정리

FK를 걸지 않는 건 "귀찮아서"가 아니라 **명확한 이유가 있는 설계 결정**인 경우가 많습니다.

- 소규모 단일 DB라면 FK가 주는 안전망이 도움이 됩니다.
- 대규모 트래픽, MSA, 샤딩 환경이라면 FK 없이 애플리케이션 레벨에서 관리하는 것이 현실적입니다.
- 어느 쪽이든 **"무결성 책임을 어디서 질 것인가"** 를 명확히 하는 것이 중요합니다.

DB 구조를 처음 설계할 때, 혹은 레거시를 리팩토링할 때 이 기준을 기억해두시면 도움이 될 것입니다.
